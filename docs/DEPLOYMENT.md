# Деплой H-Sport на VPS — пошаговая инструкция

Инструкция для развёртывания интернет-магазина на своём сервере. Написана так, чтобы ею мог воспользоваться человек **без опыта в программировании**: все команды даны целиком, после каждой команды написано, что вы должны увидеть и что делать дальше.

**Важно:** команды нужно вводить **точно** как написано (кроме тех мест, где указано «замените на своё»). Регистр букв и пробелы имеют значение.

---

## Что вам понадобится

- Компьютер с Windows (подойдёт встроенная программа «Терминал» или PowerShell).
- Доступ в интернет.
- Карта для оплаты VPS и домена (если домена ещё нет).
- 30–60 минут времени на первый раз.

## Часть 1. Заказ сервера (VPS) и домена

### Шаг 1.1. Выбор VPS

**VPS** — это арендованный компьютер в интернете, на котором будет работать ваш сайт.

Откройте сайт любого провайдера из таблицы и закажите сервер с такими параметрами:

| Параметр | Что выбрать |
| -------- | ----------- |
| **ОС** | Ubuntu 22.04 LTS (обязательно) |
| **Оперативная память (RAM)** | минимум 2 ГБ, лучше 4 ГБ |
| **Процессор (CPU)** | 1 или 2 ядра |
| **Диск** | минимум 20 ГБ SSD, лучше 40 ГБ |

Почему минимум 2 ГБ RAM: сборка сайта (команда `npm run build`) требует много памяти. На 1 ГБ сборка может упасть с ошибкой.

**Примеры провайдеров (Россия):**

- **Timeweb Cloud** — от ~300 ₽/мес, удобная панель.
- **Selectel** — от ~300 ₽/мес, надёжные дата-центры.
- **Reg.ru** — от ~200 ₽/мес.
- **Vscale** — от ~300 ₽/мес.

**Примеры зарубежных:** DigitalOcean (от $6/мес), оплата в долларах.

### Шаг 1.2. Данные сервера

Сервер заказан в **Timeweb Cloud**. Данные для подключения:

| Параметр | Значение |
|----------|----------|
| **IPv4** | `62.113.44.100` |
| **IPv6** | `2a03:6f00:a::1:70be` |
| **SSH** | `ssh root@62.113.44.100` |
| **Домен** | `h-brand.ru` (зарегистрирован в Timeweb) |

**Закрытые порты на сервере:** 25, 587, 2525, 53413, 389, 3389, 465.

Это почтовые порты (SMTP). Отправлять письма напрямую с сервера нельзя — нужен внешний SMTP-сервис (Яндекс 360, Mail.ru для бизнеса и т.п.), который указывается в `.env` в переменных `SMTP_*`.

### Шаг 1.3. Домен

Домен `h-brand.ru` добавлен в панели Timeweb. После того как сайт заработает по IP, нужно привязать домен и включить SSL — подробности в **Части 8**.

## Часть 2. Подключение к серверу с вашего компьютера

«Подключиться к серверу» значит открыть на своём компьютере текстовое окно (терминал), через которое вы будете отправлять команды на сервер.

### Шаг 2.1. Открыть терминал (Windows)

1. Нажмите клавишу **Win** (клавиша с эмблемой Windows).
2. Введите **PowerShell** или **Терминал**.
3. Запустите приложение **Windows PowerShell** или **Терминал**.

Появится окно с чёрным или синим фоном и строкой, где мигает курсор. Это и есть «терминал». Сюда вы будете вставлять команды.

### Шаг 2.2. Подключиться по SSH

В терминале введите одну строку:

```bash
ssh root@62.113.44.100
```
- Нажмите **Enter**.

**Что может произойти:**

- Появится вопрос про «fingerprint» вида:  
  `Are you sure you want to continue connecting (yes/no)?`  
  Напечатайте **yes** и нажмите **Enter**.
- Затем попросят **пароль**. Введите пароль от сервера (при вводе символы не отображаются — это нормально) и нажмите **Enter**.

**Если всё верно:** в начале строки появится что-то вроде `root@название-сервера:~#`. Значит, вы **на сервере**. Дальше все команды из разделов 3–7 нужно выполнять **уже на сервере** в этом же окне.

**Если пишет «Connection refused» или «timed out»:** проверьте, что IP верный и что у провайдера открыт порт 22 для SSH. В панели VPS обычно есть кнопка «Консоль» или «VNC» — можно попробовать войти через браузер.

## Часть 3. Установка программ на сервере (один раз)

Все команды ниже выполняются **на сервере** (после того как вы вошли по SSH). Вводите по одной строке, нажимайте **Enter**, дождитесь окончания выполнения, затем вводите следующую.

### Шаг 3.1. Обновить систему

Скопируйте и вставьте по очереди:

```bash
sudo apt update
```

Нажмите **Enter**. Появится список обновлений. Затем:

```bash
sudo apt upgrade -y
```

Подождите, пока всё установится (может занять несколько минут). Ошибок быть не должно.

### Шаг 3.2. Установить Node.js 20

Node.js — это среда, в которой работает сайт.

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
```

Затем:

```bash
sudo apt install -y nodejs
```

Проверка:

```bash
node -v
```

Должно вывести что-то вроде **v20.x.x**. Если версия 20 — всё верно.

### Шаг 3.3. Установить Git

Git нужен, чтобы скачивать код проекта с GitHub.

```bash
sudo apt install -y git
```

Проверка:

```bash
git --version
```

Должна отобразиться версия Git (например `git version 2.x.x`).

### Шаг 3.4. Установить PM2

PM2 — программа, которая будет держать ваш сайт запущенным и перезапускать его после перезагрузки сервера.

```bash
sudo npm install -g pm2
```

Проверка:

```bash
pm2 -v
```

Должна показаться версия PM2 (например `5.x.x`).

### Шаг 3.5. Установить PostgreSQL (база данных)

В базе данных хранятся товары, заказы, пользователи.

```bash
sudo apt install -y postgresql postgresql-contrib
sudo systemctl enable postgresql
sudo systemctl start postgresql
```

Ошибок быть не должно.

Теперь создаём пользователя и базу для сайта. **Замените `ВАШ_ПАРОЛЬ` на свой надёжный пароль** (латиница, цифры, знаки). Запомните этот пароль — он понадобится для файла `.env`.

```bash
sudo -u postgres psql -c "CREATE USER hsport WITH PASSWORD 'ВАШ_ПАРОЛЬ';"
```

(Подставьте свой пароль вместо `ВАШ_ПАРОЛЬ`, оставьте кавычки.)

```bash
sudo -u postgres psql -c "CREATE DATABASE hsport OWNER hsport;"
```

Если пароль уже использовался (пользователь уже есть), может появиться сообщение об ошибке «already exists». Тогда просто переходите к следующему шагу — база уже создана.

### Шаг 3.6. Настройка файрвола (UFW)

Файрвол закрывает порты сервера от внешнего доступа. Это **обязательный** шаг для безопасности — без него порт 3000 (Next.js) и другие сервисы могут быть доступны напрямую из интернета.

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp       # SSH (стандартный порт)
sudo ufw allow 80/tcp       # HTTP
sudo ufw allow 443/tcp      # HTTPS
sudo ufw deny 3000/tcp      # Блокируем прямой доступ к Next.js
sudo ufw enable
```

Если используете нестандартный порт SSH (например, 2222), замените `22/tcp` на свой порт.

Проверка:

```bash
sudo ufw status
```

Должны быть видны правила: SSH (22), HTTP (80), HTTPS (443) — ALLOW. Порт 3000 — DENY. Весь остальной входящий трафик — по умолчанию заблокирован.

## Часть 5. Настройка переменных окружения (файл .env)

Файл `.env` хранит пароли и настройки сайта (база данных, секреты, URL). Его нельзя публиковать в интернет — только хранить на сервере.

### Шаг 5.1. Создать файл .env

Выполните:

```bash
nano ~/h-sport/.env
```

Откроется простой текстовый редактор **nano** внутри терминала.

### Шаг 5.2. Вставить настройки

В проекте на GitHub есть файл **.env.example** — в нём перечислены все переменные. Вам нужно создать на сервере файл `.env` и заполнить в нём **реальные** значения.

Ниже — полный шаблон для production. Скопируйте блок в nano и замените значения на свои:

```env
# ============================================
# H-Sport — Production .env
# ============================================

# --- ОБЯЗАТЕЛЬНЫЕ ---
DATABASE_URL="postgresql://hsport:ВАШ_ПАРОЛЬ_ОТ_БАЗЫ@localhost:5432/hsport?schema=public"
NODE_ENV="production"
AUTH_SECRET="ВСТАВЬТЕ_СЮДА_РЕЗУЛЬТАТ_openssl_rand_-base64_32"
NEXT_PUBLIC_APP_URL="https://h-brand.ru"
AUTH_URL="https://h-brand.ru"

# --- АДМИНИСТРАТОР ---
# Email, который автоматически получает роль ADMIN при входе
ADMIN_EMAIL="jaksan37@gmail.com"

# --- ПОЧТА (SMTP) ---
# Без SMTP не будут работать: подтверждение email, сброс пароля, письмо о заказе
SMTP_HOST="smtp.example.com"
SMTP_PORT="465"
SMTP_USER="ваш-smtp-логин"
SMTP_PASSWORD="ваш-smtp-пароль"
SMTP_FROM="H-Sport <noreply@h-brand.ru>"

# --- CDEK API (доставка) ---
CDEK_CLIENT_ID="ваш-cdek-client-id"
CDEK_CLIENT_SECRET="ваш-cdek-client-secret"
CDEK_IS_TEST="false"
CDEK_FROM_CITY_CODE="137"

# --- DaData API (подсказки адресов) ---
DADATA_TOKEN="ваш-dadata-токен"

# --- ROBOKASSA (оплата) ---
ROBOKASSA_MERCHANT_LOGIN="ваш-логин"
ROBOKASSA_PASSWORD_1="ваш-пароль-1"
ROBOKASSA_PASSWORD_2="ваш-пароль-2"
ROBOKASSA_IS_TEST="false"
ROBOKASSA_HASH_ALGORITHM="sha256"

# --- БЕЗОПАСНОСТЬ ---
# Укажите реальный домен сайта. НЕ используйте ALLOW_ANY_ORIGIN в production!
ALLOWED_ORIGINS="https://h-brand.ru"

# --- ЛОГИРОВАНИЕ ---
LOG_LEVEL="info"
PRISMA_LOG_QUERIES="false"
ENABLE_FILE_LOGGING=true
```

**Что заменить:**

- **ВАШ_ПАРОЛЬ_ОТ_БАЗЫ** — тот же пароль, что вы задали для пользователя `hsport` в шаге 3.5.
- **AUTH_SECRET** — случайная строка не короче 32 символов. **Без неё приложение не запустится.** Сгенерировать на сервере:  
  ```bash
  openssl rand -base64 32
  ```  
  Скопируйте вывод и вставьте в `AUTH_SECRET="..."`.
- **NEXT_PUBLIC_APP_URL** и **AUTH_URL** — адрес сайта. Пока домена нет — укажите `http://ВАШ_IP:3000`. После настройки домена и SSL — `https://h-brand.ru`.

  **Важно для ссылок в письмах:** именно **AUTH_URL** (или NEXT_PUBLIC_APP_URL) подставляется в ссылки подтверждения email и сброса пароля. Если не задать их или указать localhost — ссылки в письмах будут вести на localhost и не откроются у пользователя. На сервере 85.239.34.207 укажите, например: `AUTH_URL="http://85.239.34.207:3000"` (или ваш порт/домен).
- **SMTP_***, **CDEK_***, **DADATA_TOKEN**, **ROBOKASSA_*** — получите у соответствующих сервисов. Без них соответствующие функции (почта, доставка, подсказки, оплата) не будут работать, но сайт запустится.
- **ADMIN_EMAIL** — email администратора (`jaksan37@gmail.com`). Этот пользователь автоматически получает права админа при входе.

### Шаг 5.3. Сохранить и выйти из nano

- Нажмите **Ctrl+O** (сохранить), затем **Enter** (подтвердить имя файла).
- Нажмите **Ctrl+X** (выйти из редактора).

Файл `.env` создан. Больше его никому не показывайте и не загружайте в GitHub.

## Часть 6. Первая сборка и запуск сайта

Все команды выполняются **на сервере**, в каталоге проекта.

### Шаг 6.1. Перейти в каталог проекта

```bash
cd ~/h-sport
```

### Шаг 6.2. Установить зависимости

```bash
npm ci
```

Эта команда скачивает библиотеки проекта. Может занять 1–3 минуты. В конце не должно быть красных строк с ошибками.

### Шаг 6.3. Подготовить базу данных (Prisma)

```bash
npx prisma generate
```

Затем применить миграции (создать таблицы в базе):

```bash
npx prisma migrate deploy
```

Должно вывести что-то вроде «Applied migration ...». Ошибка «no migration found» обычно значит, что миграции ещё не добавлены в репозиторий — тогда этот шаг можно пропустить, если разработчик не требовал иного.

### Шаг 6.4. Собрать проект

```bash
npm run build
```

**Важно:** сборка может занять 3–10 минут и потребляет много памяти. На сервере с 2 ГБ RAM дождитесь окончания. В конце должно быть сообщение вроде «Compiled successfully» или «Route (app) ...».

Если сборка падает с ошибкой про память — перезапустите сервер и снова выполните только `npm run build` (остальное уже сделано).

Скопируйте статику и папку сайта в служебный каталог (без этого картинки и стили не будут загружаться):

```bash
cp -r .next/static .next/standalone/.next/
cp -r public .next/standalone/
```

**Фото товаров:** загруженные в админке изображения сохраняются в `public/assets/images/products/` и отдаются через API (запросы к `/assets/images/products/*` перенаправляются на него). Важно запускать приложение из **корня проекта** (как в ecosystem.config.cjs: `cwd` — корень), чтобы загрузки и раздача шли из одной и той же папки `public/`.

### Шаг 6.5. Создать каталог для логов

```bash
mkdir -p logs
```

### Шаг 6.6. Запустить сайт через PM2

```bash
pm2 start ecosystem.config.cjs
```

Должно появиться сообщение о том, что приложение `h-sport` запущено. Сохраняем список процессов PM2, чтобы после перезагрузки сервера сайт поднялся сам:

```bash
pm2 save
```

Затем включите автозапуск PM2 при загрузке сервера:

```bash
pm2 startup
```

В конце команда выведет **ещё одну команду** (начинается с `sudo env ...`). Эту команду нужно **скопировать и выполнить** — она один раз настраивает автозапуск. После выполнения снова нажмите **Enter**.


На **своём компьютере** откройте браузер и перейдите по адресу:

Должна открыться главная страница магазина. Можно также проверить:

## Часть 8. Домен, DNS и SSL

Когда сайт уже работает по IP (проверили в части 7), привязываем домен `h-brand.ru` и включаем HTTPS.

### Шаг 8.1. DNS-записи в Timeweb

Домен `h-brand.ru` уже добавлен в панели Timeweb. Нужно настроить DNS-записи, чтобы домен указывал на сервер.

В панели Timeweb: **Домены → h-brand.ru → DNS-записи**. Настройте:

| Тип | Имя | Значение |
|-----|-----|----------|
| A | `@` | `62.113.44.100` |
| A | `www` | `62.113.44.100` |

Если там есть старые записи от Timeweb (паркинг) — удалите их, оставьте только свои.

После сохранения DNS обновляются от нескольких минут до 24 часов (обычно 15–60 мин). Проверить можно командой:

```bash
ping h-brand.ru
```

Когда в ответе появится `62.113.44.100` — DNS заработал.

### Шаг 8.2. Nginx (проксирование)

Nginx нужен как «посредник» между интернетом и приложением на порту 3000: он принимает запросы на порт 80/443 и передаёт их приложению.

```bash
sudo apt install nginx -y
```

Создайте конфиг:

```bash
sudo nano /etc/nginx/sites-available/h-brand.ru
```

Вставьте:

```nginx
server {
    listen 80;
    server_name h-brand.ru www.h-brand.ru;

    # Скрываем версию Nginx из заголовка Server
    server_tokens off;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        # Скрываем заголовок X-Powered-By
        proxy_hide_header X-Powered-By;
    }
}
```

Включите конфиг и перезапустите Nginx:

```bash
sudo ln -s /etc/nginx/sites-available/h-brand.ru /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl restart nginx
```

Если `nginx -t` вывел `syntax is ok, test is successful` — всё верно. Теперь `http://h-brand.ru` должен открывать сайт (пока без HTTPS).

### Шаг 8.3. SSL-сертификат (бесплатный Let's Encrypt)

**Когда делать:** сразу после того, как `http://h-brand.ru` открылся в браузере. Не откладывайте — без SSL браузеры показывают «Не защищено», а Robokassa и другие интеграции требуют HTTPS.

**Вариант А — через Timeweb (проще):**

В панели Timeweb: **Домены → h-brand.ru → SSL-сертификат** → выпустить бесплатный Let's Encrypt. Подождите 1–5 минут, проверьте `https://h-brand.ru`.

**Вариант Б — через Certbot на сервере (если вариант А не подходит):**

```bash
sudo apt install certbot python3-certbot-nginx -y
sudo certbot --nginx -d h-brand.ru -d www.h-brand.ru
```

Certbot задаст пару вопросов (email для уведомлений, согласие с условиями). После этого сертификат установится автоматически и будет продлеваться сам.

Проверьте: откройте `https://h-brand.ru` — должен быть замочек в адресной строке.

### Шаг 8.4. После подключения SSL

Обновите `.env` на сервере:

```bash
nano ~/h-sport/.env
```

Измените:

```
NEXT_PUBLIC_APP_URL=https://h-brand.ru
ALLOWED_ORIGINS=https://h-brand.ru
```

Перезапустите приложение:

```bash
pm2 restart h-sport --update-env
```

Также проверьте:
- В ЛК **Робокассы** — все URL (Result, Success, Fail) должны начинаться с `https://h-brand.ru/...`
- Сайт по `http://h-brand.ru` должен автоматически перенаправлять на `https://` (Certbot обычно настраивает это сам).

## Часть 9. Обновление сайта после изменений в коде

Когда разработчик (или вы) внёс изменения в код и залил их на GitHub, нужно обновить сайт на сервере.

### Шаг 9.1. Подключиться к серверу

Как в части 2: откройте PowerShell/терминал и выполните:

```bash
ssh root@62.113.44.100
```

### Шаг 9.2. Запустить скрипт обновления

```bash
cd ~/h-sport
./scripts/server-deploy.sh
```

Скрипт сам:

- подтянет последние изменения с GitHub;
- установит зависимости;
- применит миграции базы данных;
- пересоберёт проект;
- перезапустит приложение в PM2.

Дождитесь сообщения **Deploy done**. После этого обновлённый сайт уже работает.

**База данных при обновлении не перезатирается:** скрипт деплоя не запускает сидирование (`prisma seed`). Применяются только миграции схемы (новые таблицы/поля), все товары, заказы и настройки на сервере остаются как есть. Команду `npm run prisma:seed` на сервере запускайте только один раз при первой установке (если нужно заполнить базу из файлов), потом не трогайте — иначе можно получить дубликаты товаров.

**Если репозиторий на ветке `main`, а не `master`:** перед запуском скрипта выполните:

```bash
export BRANCH=main
./scripts/server-deploy.sh
```

Либо один раз переименуйте ветку в репозитории на GitHub в `master`, тогда скрипт будет работать без изменений.

### Запуск деплоя автоматически при пуше в GitHub

В проекте есть GitHub Actions (файл `.github/workflows/deploy.yml`): при каждом **пуше в ветку master** он подключается к серверу по SSH и запускает `./scripts/server-deploy.sh`. То есть все шаги (pull, зависимости, миграции, сборка, перезапуск) выполняются без вашего участия.

Чтобы это работало, нужно один раз создать SSH-ключ, добавить его на сервер и прописать секреты в GitHub. Ниже — пошагово.

#### Шаг A. Создать SSH-ключ на своём компьютере

1. Откройте **PowerShell** или **Терминал** (как в части 2 инструкции).
2. Выполните команду создания ключа (без пароля — когда спросит passphrase, просто нажмите **Enter** два раза).

**В PowerShell (Windows):**

```powershell
ssh-keygen -t ed25519 -C "github-deploy" -f "$env:USERPROFILE\.ssh\github_deploy" -N '""'
```

**В Git Bash / WSL / Linux:**

```bash
ssh-keygen -t ed25519 -C "github-deploy" -f "$HOME/.ssh/github_deploy" -N '""'
```

Если программа спросит «Enter passphrase», просто нажмите **Enter** (и ещё раз для подтверждения).

Если папки `.ssh` нет, её создадут. Появится два файла:

- `github_deploy` — **приватный** ключ (его содержимое потом вставите в секрет GitHub).
- `github_deploy.pub` — **публичный** ключ (его нужно добавить на сервер).

3. Посмотреть публичный ключ одной строкой (он понадобится на сервере):

**Windows PowerShell:**

```powershell
Get-Content $env:USERPROFILE\.ssh\github_deploy.pub
```

**Git Bash / WSL / Linux:**

```bash
cat ~/.ssh/github_deploy.pub
```

Скопируйте всю строку (начинается с `ssh-ed25519`, заканчивается `github-deploy`). Пока не закрывайте окно — ключ понадобится на следующем шаге.

#### Шаг B. Добавить публичный ключ на сервер

1. Подключитесь к серверу по SSH (как в части 2), например:  
   `ssh root@62.113.44.100`
2. На сервере выполните по очереди:

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "СЮДА_ВСТАВЬТЕ_СКОПИРОВАННУЮ_СТРОКУ_ИЗ_github_deploy.pub" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

Вместо `СЮДА_ВСТАВЬТЕ_...` вставьте **одну строку** из вывода команды из шага A.3 (без переносов и лишних пробелов). Кавычки в команде `echo` оставьте как есть, внутри кавычек — только эта одна строка ключа.

3. Выйдите с сервера: `exit`.

#### Шаг C. Добавить секреты в GitHub

1. Откройте репозиторий на **github.com**.
2. Перейдите: **Settings** → **Secrets and variables** → **Actions**.
3. Нажмите **New repository secret** и создайте по одному секрету:

| Имя секрета   | Значение |
|---------------|----------|
| `SSH_HOST`    | IP сервера или домен (`62.113.44.100` или `h-brand.ru`). |
| `SSH_USER`    | Пользователь, под которым заходите по SSH (например `root` или `deploy`). |
| `SSH_PRIVATE_KEY` | **Полное содержимое** файла приватного ключа. На Windows в PowerShell: `Get-Content $env:USERPROFILE\.ssh\github_deploy` — скопируйте весь вывод, включая строки `-----BEGIN OPENSSH PRIVATE KEY-----` и `-----END OPENSSH PRIVATE KEY-----`. |
| `DEPLOY_PATH` (необязательно) | Абсолютный путь к каталогу проекта на сервере, если он не `/home/deploy/h-sport`. Например `/root/h-sport` или `/home/deploy/h-sport`. |

После сохранения секретов при следующем **пуше в ветку master** GitHub сам подключится к серверу и выполнит деплой. Проверить можно во вкладке **Actions** репозитория: должен появиться запуск workflow «Deploy to production».

Если что-то пойдёт не так (например, «Permission denied»), проверьте: публичный ключ действительно добавлен в `~/.ssh/authorized_keys` на сервере у пользователя `SSH_USER`, а в секрет `SSH_PRIVATE_KEY` вставлен именно приватный ключ целиком, без лишних пробелов в начале и в конце.

## Часть 10. Что нельзя публиковать и куда не лезть

- Файл **.env** — в нём пароли и ключи. Никогда не загружайте его в GitHub и никому не отправляйте. В репозиторий можно коммитить только **.env.example** (шаблон без секретов).
- Не удаляйте и не меняйте случайно каталоги **.next**, **node_modules**, **prisma** и файл **ecosystem.config.cjs** — без них сайт не запустится.
- Не выполняйте на сервере команды, которые вам прислали незнакомые люди или которые вы нашли на сомнительных сайтах — так можно сломать сервер или потерять данные.

## Часть 11. Полезные команды PM2 (шпаргалка)

Выполняются **на сервере** после `ssh ...` и при необходимости после `cd ~/h-sport`.

| Действие | Команда |
| -------- | ------- |
| Посмотреть, запущен ли сайт | `pm2 status` |
| Посмотреть логи в реальном времени | `pm2 logs h-sport` |
| Перезапустить сайт (после смены .env) | `pm2 restart h-sport --update-env` |
| Остановить сайт | `pm2 stop h-sport` |
| Снова запустить сайт | `pm2 start ecosystem.config.cjs` |

Выход из просмотра логов: **Ctrl+C**.
